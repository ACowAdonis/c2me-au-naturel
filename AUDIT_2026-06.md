# C2ME-au-naturel Full Audit — June 2026

Scope: all 22 modules plus fork-specific diff (9e55412..HEAD). Method: five parallel
deep reviews (fork diff adversarial review; c2me-base; chunk I/O stack; worldgen stack;
remaining modules), with all CRITICAL and most HIGH findings independently re-verified
against source line-by-line. Findings marked [V] were directly verified; [A] are
agent-reported with quoted evidence but not independently re-read.

Only material findings are included: incorrect behavior, data loss, deadlock/starvation,
leaks, or measurable performance impact. Style and cold-path issues are omitted.

---

## Commentary

### Overall state

The codebase's core architectural choices are sound: neighbor-locking worldgen state and
both priority queues are correctly confined to the single `c2me-sched` thread, the
notickvd fix (8e9c3b6) is correct and well-reasoned, and the BitSet
`DynamicPriorityQueue` rewrite (5988919) is verified correct for its single-threaded
usage. The async catcher (dc450ec) is functionally correct as a tripwire.

Three systemic patterns emerged, and they should inform process as much as code:

1. **Performance reorders that broke correctness invariants.** Commit dc46ac8
   ("Improve chunk loading prioritization") reordered `C2MEStorageThread.pollTasks()`
   to serve reads before intaking writes, without recognizing that `handlePendingWrites`
   is the read-consistency mechanism (it populates the write-visibility cache), not an
   I/O cost. The result is the single most dangerous defect in the fork (C1). The same
   commit's backlog gating creates write starvation (H1).

2. **Observability theater.** The fork's monitoring additions count the wrong things:
   `writeQueueSize` counts a transit queue drained every cycle, while the real
   accumulation points (`writeBacklog`, `cache`, `writeFutures`) are unbounded and
   uncounted; the RNG warning rate-limiter still pays a full stack walk on every
   off-thread call; the lock-timing warnings detect symptoms of problems this audit
   identifies directly.

3. **PERFORMANCE_AUDIT.md "implemented" statuses overstate hardening.** Issue 2.1's
   lock-free state machine (commits 63e79eb, f236794) was applied to code that was
   already dead at fork time and later deleted — net effect zero. Issue 3.5's
   `withScope` helpers have no call sites. That document should be retired or
   re-baselined against this one.

### Severity distribution

- 3 CRITICAL (chunk data rollback; worldgen throttle disabled; Perlin noise defect)
- 7 HIGH (second save-loss vector; write starvation; shutdown durability; infinite
  unwrap loop; lock leak on throw; pool-starvation deadlock; swallowed write errors)
- ~20 MEDIUM (leaks, wedges, NBT-restore regressions, determinism, races)

## CRITICAL findings

### C1 [V] Chunk data rollback: read/write reorder in storage thread (fork dc46ac8)
`c2me-rewrites-chunkio/.../C2MEStorageThread.java:130-142`

`pollTasks()` runs `handlePendingReads()` before `handlePendingWrites()`. The only
read-your-writes mechanism is `cache`, populated solely by `handlePendingWrites()`
(line 276). Interleaving: chunk unload enqueues write W; player returns and a read R
for the same pos is enqueued strictly after W; the storage thread serves R before
intaking W; `cache.get(pos)` misses; stale data is read from disk; the server holds a
pre-unload chunk which later re-saves and overwrites W. Permanent, silent rollback of
player changes, opening exactly during unload/reload churn. Upstream's order
(`handleTasks -> handlePendingWrites -> handlePendingReads`) had only a microsecond
window; the fork's order makes the window a full poll cycle.

**Fix:** restore `handlePendingWrites()` before `handlePendingReads()`. This costs
nothing — `handlePendingWrites` does no I/O; the intended read-prioritization is fully
delivered by the `writeBacklog()` deferral alone. Optionally drain
`pendingWriteRequests` at the top of each read-loop iteration to close the residual
upstream window.

### C2 [V] `scheduledCount` underflow permanently disables the worldgen concurrency cap
`c2me-base/.../scheduler/SchedulingManager.java:37-47,119,137-146`

Async tasks (`enqueue` -> `if (task.isAsync()) schedule0(task)`) never pass the
increment at line 119, but `schedule0` unconditionally installs the decrementing
post-action, which `NeighborLockingTask.runTask` always invokes. Every
`ChunkStatus.LIGHT` task is async (`MixinChunkStatus.java:134`), so the counter drifts
≈ −1 per generated chunk. `scheduledCount.get() < maxScheduled` becomes permanently
true within minutes: unbounded task admission, the `DynamicPriorityQueue` drains
immediately to the FIFO pool queue, and both level-based priority and the sync-load
boost become no-ops — the fork's chunk-prioritization feature is defeated at runtime.
The dead `SCHEDULED_ASYNC` enum constant marks the unfinished intent.

**Fix:** skip the decrement for async tasks (e.g. in `schedule0`:
`if (!task.isAsync()) scheduledCount.decrementAndGet();` inside the post-action), or
increment symmetrically on the async path. Note: all current throughput observations
were measured with the cap disabled; re-benchmark after fixing.

### C3 [V] Hash-based Perlin sampler: period-16 tiling and loss of vanilla parity
`c2me-opts-math/.../MixinPerlinNoiseSampler.java:78-92`

`gradDot` selects gradients via `hash & 15`. Since `(a*b) mod 16` depends only on
`a mod 16`, the gradient index is a pure function of `x mod 16, y mod 16, z mod 16,
seed mod 16`: the gradient field tiles exactly every 16 lattice cells in every axis —
structured, repeating artifacts in terrain, caves, and density functions. The fork's
own benchmark implements this correctly (`src/jmh/java/noise/PerlinNoiseBenchmark.java:226`:
`((hash >> 15) & 15) << 2`), so the shipped mixin is not the variant that was
benchmarked. Independently of the bug, the hash approach abandons bit-exact parity
with vanilla worldgen (chunk-border discontinuities against previously generated
terrain, no seed parity); upstream C2ME's opts-math is bit-exact. `hashSeed` also
truncates the permutation table to 32 bits.

**Fix:** vanilla parity is explicitly not a goal for this fork, so keep the hash
sampler and apply `((hash >>> 15) & 15) << 2` — this makes the deployed code identical
to the benchmarked b3 variant (FastNoiseLite's production construction). While in the
file: widen `hashSeed` to 64 bits (currently truncates the permutation table to 4
bytes), check whether b4 (FMA) or b5 (table-free gradients) benchmarked better and use
the winner, and do a one-off quality check (visual terrain pass or gradient-index
histogram over a large lattice volume). Note: any change to gradient selection seams
against chunks generated by the current broken sampler; this cost is unavoidable and
identical for all fix options.

## HIGH findings

### H1 [V] Write-backlog starvation under sustained read load (fork dc46ac8)
`C2MEStorageThread.java:137-140,343-361`

`writeBacklog()` runs only when `pendingReadRequests.isEmpty()`. Reads do synchronous
region-file I/O on the storage thread, so during active exploration the read queue can
stay non-empty for long stretches — zero writes reach disk for the duration.
`writeBacklog` + `cache` retain full NBT for every dirty chunk (unbounded heap growth,
uncounted by any monitor), and a crash during the window loses every backlogged chunk.

**Fix:** make the gate a throttle: always write 1-2 backlog entries per cycle even with
reads pending; bound the backlog with a forced-flush threshold.

### H2 [V] `asyncSave` clears `needsSaving` at schedule time; every retry/fallback path is a no-op
`c2me-threading-chunkio/.../MixinThreadedAnvilChunkStorage.java:325-328,353-357,382-385,400-413`
(upstream-inherited)

`setNeedsSaving(false)` runs before the save is durable. All three recovery paths —
deferred re-entry when the saving future is incomplete (355), re-schedule on
`TaskCancellationException` (383), and "falling back to sync saving" (409-411, vanilla
`save` has the identical guard) — re-enter through `if (!chunk.needsSaving()) return
false` and silently save nothing. Since `tryUnloadChunk` discards the in-memory chunk
regardless, any async serialization failure or saving-future race reverts the chunk to
its last on-disk state.

**Fix:** `chunk.setNeedsSaving(true)` before any deferral/fallback re-entry, or move
the clear to after successful `setNbt`/`setRawChunkData` submission (guarding against
concurrent re-dirtying).

### H3 [V] Infinite loop unwrapping `CompletionException`
`c2me-threading-worldgen/.../MixinChunkStatus.java:137-146`

`while (actual instanceof CompletionException) actual = t.getCause();` — re-reads
`t.getCause()` instead of `actual.getCause()`. A double-wrapped `CompletionException`
pins the completing thread (a worker or the `c2me-sched` thread) in a hot infinite
loop, permanently losing the thread and potentially hanging chunk generation.

**Fix:** `actual = actual.getCause()`, treating `null` as terminal.

### H4 [V] `NeighborLockingTask`: lock leak on synchronous throw; cancellation never honored
`c2me-base/.../scheduler/NeighborLockingTask.java:19,56-78`

(a) `runTask` calls `this.action.get()` with no try/catch after `tryPrepare()` has
acquired the full (2r+1)^2 lock region. A synchronous throw (reachable: `AS_IS`
statuses run `doWork` synchronously, including any modded `ChunkStatus`) leaks every
lock, never runs the post-action (slot leak), and never completes `this.future` —
permanent generation deadlock for the area. (b) The `isCancelled` field is stored and
never read; `ChunkStatusUtils.runChunkGenWithLock` builds a real holder-downgrade
predicate that is dead. Unloaded/downgraded chunks still acquire full lock sets and run
complete generation — wasted CPU plus contention on every unload race.

**Fix:** try/catch around `action.get()`: release all names, run the post-action,
`completeExceptionally`. Check `isCancelled.getAsBoolean()` at the head of
`tryPrepare()`; on true, complete with `CancellationException` (the `MixinChunkStatus`
handler already maps it to `UNLOADED_CHUNK`).

### H5 [A] `CFUtil.join` interrupt busy-spin; pool-starvation deadlock reachable
`c2me-base/.../util/CFUtil.java:8-13`

`LockSupport.parkNanos` returns immediately once the thread's interrupt flag is set and
does not clear it — after any interrupt the loop becomes a 100% CPU spin until the
future completes (possibly never during shutdown). Even uninterrupted: ~10k wakeups/sec
per blocked thread. Worse: `c2me-opts-chunk-access` workers block here on chunk-load
futures whose completion requires worldgen tasks on the same fixed-size
`GlobalExecutors.executor` — if all N workers block simultaneously, permanent deadlock.
The commented-out ForkJoinPool in `GlobalExecutors.java:19-24` would have provided
helping; the current pool does not.

**Fix:** handle interrupt (check/restore, fall through to `future.join()`); medium-term,
use a `ManagedBlocker`-style or helping approach for off-thread chunk waits.

### H6 [A] Storage `close()` does not join; daemon thread can be killed mid-flush
`C2MEStorageVanillaInterface.java:47-49`, `C2MEStorageThread.java:84`

Vanilla `StorageIoWorker.close()` blocks until durable. The replacement fires `closing`
and returns; the flushing thread is a daemon, so JVM exit can kill it mid-write.
Anything enqueued between the last `completeAll` and `close()` (late POI/entity writes,
H2's re-scheduled saves) has no barrier.

**Fix:** `this.backend.close().join()` in `close()`; consider a non-daemon thread.

### H7 [A] Failed region writes: error swallowed, stale cache retained, flush reports success
`C2MEStorageThread.java:449-468,371-374`

On write failure `SneakyThrow.sneaky` aborts before `cache.remove(pos)` (permanent
stale entry + memory) and `flushBacklog` maps every future through
`.exceptionally(unused -> null)`, so `flush(true)`/`completeAll` complete normally while
the chunk never reached disk — the server reports saved; the on-disk chunk is stale.

**Fix:** on failure re-insert into `writeBacklog` with bounded retries; propagate
persistent failure out of `flush`; remove from cache only on success.

### H8 [REFUTED] Spawn-blacklist redirect returns null into an unguarded vanilla path
`c2me-opts-spawning/.../MixinSpawnHelperPopulate.java:27-32`

WITHDRAWN after decompile verification (2026-06-12): the 1.20.1 bytecode of
`SpawnHelper.populateEntities` (`NaturalSpawner.spawnMobsForChunkGeneration`)
null-checks the `EntityType.create` result immediately after the call
(`ifnonnull` -> proceed; null -> `continue` to the loop bottom), identically in both
cached named jars. Returning null is safe and is the cleanest blacklist
implementation available — no change required. The original agent finding was based
on pre-1.19.3 code shape and was flagged for decompile verification, which refuted it.

## MEDIUM findings

### Scheduler (become live once C2 is fixed)
- **M1 [V] Lost wakeup** — `SchedulingManager.java:114-124`: drain loop exits at cap; a
  completing worker's `scheduleExecution()` CAS fails while `scheduled` is still true;
  `scheduled.set(false)` follows with no re-check — queued tasks stranded until the next
  external event. Fix: re-check queue/slots after clearing the flag and retry the CAS.
- **M2 [V] Head-of-line blocking** — `SchedulingManager.java:118,126-135`:
  `NOT_SCHEDULED` from a lock-blocked head task terminates the whole drain despite free
  slots and runnable tasks elsewhere. Fix: continue the loop on `NOT_SCHEDULED`.
- **M3 [A] Single global scheduler thread, undocumented confinement invariant** — all
  dimensions share `asyncScheduler`; the correctness of every unsynchronized scheduler
  structure depends on it being single-threaded with nothing enforcing that; `AS_IS`
  statuses execute `doWork` synchronously on it, stalling scheduling in all dimensions.
  Fix: assert thread identity; dispatch `AS_IS` work to the worker pool.

### Storage / serialization
- **M4 [A] Monitoring counts the wrong queue** — `C2MEStorageThread.java:47-50`:
  `writeQueueSize` measures the promptly-drained transit queue; `writeBacklog` +
  `writeFutures` (the real accumulation) are unbounded and uncounted; no backpressure
  anywhere. Fix: count the real accumulation points; bound in-flight writes.
- **M5 [A] Shutdown windows** — `C2MEStorageThread.java:160-229`: reads enqueued during
  the closing race hang forever (the 60s `orTimeout` is on a derived future and only
  logs); writes after thread exit are silently dropped; `flush()` after exit deadlocks.
  Fix: reject operations after `closing`; final drain after `closing` is observed.
- **M6 [A] `completeAll` snapshot misses re-scheduled saves** —
  `threading-chunkio MixinThreadedAnvilChunkStorage.java:447-452`: retries scheduled
  during the wait are not awaited. Fix: loop until no incomplete futures remain.
- **M7 [A] Starlight path emits orphan light tags** —
  `chunk-serializer ChunkDataSerializer.java:421-471`: light-only boundary sections
  write nibble tags without opening a compound (`hasInner == false`) — the tags merge
  into the next section's compound or the root; boundary light state silently lost on
  every save. Fix: restore the `compoundEntryStart()` guards.
- **M8 [A] `writeTicks` count/iterate mismatch under concurrent mutation** —
  `ChunkDataSerializer.java:666-699`: list header count taken separately from
  iteration; concurrent tick-queue mutation yields structurally corrupt NBT. Fix:
  count entries actually written.
- **M9 [A] Scope-stack "recovery" corrupts unrelated in-flight serializations** —
  `AsyncSerializationManager.java:45-77`: depth-overflow/mismatch does `stack.clear()`,
  destroying other serializations' scopes on the same pooled thread; their `getScope`
  then returns null and serialization silently falls back to live data mid-save. The
  `withScope` helpers (115-138) are dead code. Fix: fail only the offending save; never
  clear; delete the dead helpers.
- **M10 [A] `NbtWriter` native-memory leak on failure** — `chunk-serializer TheMod.java:13-21`:
  no try/finally around `write` (the sync-path mixin gets this right). Fix: try/finally.
- **M11 [A] `MixinStorageIoWorker.setRawChunkData` completes a pending vanilla write's
  future without removing it** — `c2me-base/.../MixinStorageIoWorker.java:35-50`:
  consumers observe durability before the write executes; possible inverted ordering
  with the still-queued stale write. Fix: remove/supersede the entry, or don't complete
  the foreign future.
- **M12 [A] `getNeedBlending` is timing-dependent** — `MixinProtoChunk.java:66-75`: the
  fork's "return true if not yet computed" makes blending decisions depend on disk
  latency — same seed, different terrain at old-chunk borders. Fix: compose
  `blendingComputeFuture` into the consuming status future.
- **M13 [A] Deferred `onBlockChanged` callbacks run LIFO** — `MixinChunkRegion.java:21-32`:
  POI add/remove for the same position can apply out of order (portals, beds, job
  sites). Fix: chain sequentially or drain an ordered list.

### Worldgen fidelity
- **M14 [A] `StructureLocator` inner maps unsynchronized vs locked eviction** —
  `MixinStructureChecker.java:40-54`: vanilla mutates the inner `Long2BooleanOpenHashMap`
  outside `mapMutex` while the redirected eviction removes under it — put/remove race on
  an open-hash map. Fix: synchronize the inner maps on the same mutex.
- **M15 [A] `MixinStructureStart` drops deserialized reference counts** — all reads
  redirected to `referencesAtomic` (initial 0) but the NBT-load constructor's write is
  not mirrored; counts reset to 0 across reload and persist as 0. Fix: seed the atomic
  at ctor RETURN.
- **M16 [A] `MixinDesertTempleGenerator` loses restored `hasPlacedChest` flags** —
  `compareAndSet(index, false, value)` fails against initial `null`; a piece saved with
  the chest placed reloads as unplaced — loot duplication on resumed generation. Fix:
  pre-fill `FALSE` or redirect ctor writes to plain `set`.
- **M17 [A] `SimplifiedAtomicSimpleRandom` ctor skips the vanilla seed scramble** —
  `(seed ^ MULTIPLIER) & SEED_MASK` applied in `setSeed` but not the constructor; any
  redirected site consuming the ctor-seeded sequence without re-seeding diverges from
  vanilla seeds. DOWNGRADED: pure vanilla-parity issue and parity is not a goal for
  this fork; output is still deterministic per world seed. Optional one-line tidy
  (`setSeed(seed)` in the ctor), not a required fix.
- **M18 [V/A] `CheckedThreadLocalRandom` (fork 6d83e87)** — the dedup gate runs after
  `getCallerLocation()`, so every off-thread `next()` still pays a full
  `Thread.currentThread().getStackTrace()` — exactly the hot path the rate limiter was
  meant to protect; the `className.contains("Thread")` frame filter skips
  `ThreadedAnvilChunkStorage` etc., collapsing distinct call sites;
  `enforceSafeWorldRandomAccess` defaults to `false` while its comment claims fail-hard;
  once-per-site logging hides recurring desync. Fix: cheap flag/counter gate before any
  stack capture; exact-prefix filter; reconcile the default with the comment; periodic
  aggregate counts.

### notickvd / holders / misc
- **M19 [A] `NoTickSystem` wedges permanently on any exception** —
  `NoTickSystem.java:90-123`: the tick body runs unguarded before `isTicking.set(false)`;

- **M20 [A] `MixinChunkHolder.getChunkAt` placeholder leak** — `MixinChunkHolder.java:86-113`:

- **M21 [A] notickvd cross-thread reads/iteration** — `PlayerNoTickDistanceMap.java:52-56`

- **M22 [A] Sync-load priority state not cleared on all paths** —
  `c2me-base MixinServerChunkManager.java:45-68`: clear inject misses the

- **M23 [V] `MCUtil.toLong` sign-extension** — `MCUtil.java:11`: missing `& 0xFFFFFFFFL`

### Performance (hot-path, after correctness)
- **P1 [A]** `writeFutures` is `ObjectArraySet`: O(n) add + every-cycle O(n) `removeIf`
  → O(n^2) during bursts. Use `ArrayDeque`/`ObjectOpenHashSet`.
- **P2 [A]** `flushBacklog` busy-spins a full core for the duration of every autosave.
  Park or wait with wakeup.
- **P3 [A]** `c2me-fixes-chunkio-threading-issues MixinChunkHolder.beforeTick` clones
  the entire holder map per holder-tick when dirty (vanilla: once per TACS tick) —
  O(holders^2) during load/unload storms. Consider a concurrent map for async readers.
- **P4 [A]** Per-tick `ConcurrentLinkedQueue.removeIf` + O(n) `size()` monitoring in
  threading-chunkio `onTick`; throttle to every N ticks.

## Sequential remediation plan

Ordering principle: data integrity first; then scheduler correctness (whose fixes
activate latent issues in a known order); then durability; then fidelity; performance
last, because C2 means all existing benchmarks measured an uncapped scheduler and must
be redone after Phase 1.

**Phase 0 — data integrity (independent, small diffs; ship immediately)**
1. C1: restore write-intake before reads in `pollTasks` (keep backlog gating).
2. H2: re-arm `needsSaving` on all `asyncSave` retry/fallback paths.
3. H3: fix the `CompletionException` unwrap loop.
4. C3: apply the `>>>15` shift to the hash sampler (parity is not a goal); widen
   `hashSeed` to 64 bits; validate gradient distribution.
5. H8: withdrawn — decompile verification showed vanilla null-checks the create
   result; the existing null return is correct as-is.

**Phase 1 — scheduler correctness — DONE 2026-06-12 (by removal, not repair)**
Decision: the priority/admission machinery had been runtime-inert since inception
(C2's underflow held the gate permanently open), and all field experience — including
the user's workloads after moving off Distant Horizons — was accumulated without it.
Rather than repairing it, it was removed; a priority queue can be reinstated if
testing in the wild shows degraded behavior.
6. C2/M1/M2/M22: closed by deletion — SchedulingManager stripped to a neighbor-lock
   dispatcher, setLevel priority feeder and sync-load boost mixin deleted, dead
